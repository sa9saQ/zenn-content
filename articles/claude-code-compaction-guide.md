## 「さっき言ったよね？」が通じなくなる瞬間

長い会話をしていると、Claudeの動きが変わる瞬間がある。

さっきまで守っていたルールを破る。「テスト書いてから実装」って言ったのに、テストを飛ばす。ファイル構成のルールを無視する。

バグじゃない。**Compaction**が発動している。

---

## Compactionとは何か

Claude Codeのコンテキストウィンドウには、隠れたバッファがある。Opus 4.6で最大1Mに拡張されたが（2026年3月GA）、仕組みは同じ。

| 指標 | 値 |
|------|-----|
| コンテキストウィンドウ | 最大1Mトークン（Opus 4.6） |
| バッファ（使えない領域） | 約33,000トークン |
| 自動Compactionトリガー | 使用率**83.5%** |

ウィンドウが広がっても、バッファは予約されているし、83.5%でauto-compactが発動する仕組みは変わらない。1Mなら約835Kで発動する計算。余裕は増えたが、長い会話を続ければいつかは来る。

使用率が83.5%に達すると、auto-compactが発火する。何が起きるか:

1. 会話履歴が**要約**される
2. CLAUDE.mdはディスクから**再読み込み**される（生存）
3. 細かい指示は要約で**パラフレーズ**される
4. パラフレーズされた指示は**精度が落ちる**
5. Claudeは「既に議論済み」と判断してファイルを**再読みしない**

3番と4番が致命的。「テストファースト」という指示が要約で「テストを書く」に変わり、「テストを先に書く」というニュアンスが消える。そういうこと。

## 遵守率60-70%の衝撃

GitHub Issue #14258に報告されたデータ:

> Compaction後のフレームワーク遵守率: **60-70%**

10個の指示のうち3-4個が守られなくなる。しかもエラーは出ない。Claudeは普通に動き続ける。ただ、指示の一部を忘れているだけ。

前回のスキル予算問題（80個のスキルが無言で消えた）と同じ構造。**見えない劣化**。気づかないから対策もしない。対策しないから劣化が積み重なる。

---

## 33Kバッファの経緯

このバッファ、以前は**45K**だった。2026年初頭に33Kに縮小された（非公開の改善）。12K分の空き（約6%）が使えるようになった。

ユーザーの感覚としては「最近ちょっと長く使えるようになった？」くらいの変化。でも裏では12,000トークン分の改善が入ってる。

`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 環境変数でトリガー閾値を変更できるけど、下げると頻繁にCompactionが走り、上げるとギリギリまで引っ張ってから一気に大量の文脈が消える。いじらない方がいい。

---

## 7つの対策

### 1. CLAUDE.mdに「Compact Instructions」を書く

```markdown
## Compact Instructions
When compacting, always preserve: modified file paths,
test/build status, current task context,
key architectural decisions made this session.
```

Compaction時に**何を生存させるか**を指示する。短く。これだけでCompaction後の精度がかなり変わる。

### 2. 手動 `/compact`（70%時点で打つ）

自動Compaction（83.5%）を待たない。70%の時点で手動で `/compact` を打つ。

なぜか。83.5%ギリギリで発動すると、**大量の文脈を一気に要約**しなければならない。70%時点なら、まだ余裕があるうちに穏やかに要約できる。

### 3. `/clear` でタスクを切り替える

タスクが変わったら `/clear`。新鮮なコンテキストで始める。

「せっかくここまで会話したのにもったいない」と思うかもしれない。でもCompaction後の60-70%遵守率で作業を続ける方がもったいない。やり直した方が速い場合が多い。

### 4. CLAUDE.mdに永続ルールを配置

Compaction後もCLAUDE.mdは**ディスクから再読み込み**される。つまりCLAUDE.mdに書いた指示は消えない。

絶対に忘れてほしくないルールはCLAUDE.mdに書く。会話の中で「これ覚えてね」と言ったルールは消える可能性がある。

### 5. サブエージェントに重い作業を委任

調査、コード検索、大量のファイル読み込み。これらをサブエージェントに投げると、メインウィンドウのトークンが節約される。

メインウィンドウの寿命 = Compactionまでの距離。サブエージェントは独立コンテキストだから、どれだけ仕事をさせてもメインが圧迫されない。

### 6. `/context` でトークン占有を監視

どのくらいコンテキストが埋まっているか、`/context` で確認できる。StatusLineの `remaining_percentage` でも常時モニタリング可能。

「そろそろやばいかも」と感じたら確認。感覚より数字。

### 7. Compactionモデルを変更（上級者向け）

Compaction時の要約品質はモデルに依存する。デフォルトのSonnetからSonnet 1M等に変更すると、要約の質が上がる可能性がある（ただし未検証、medium信頼度）。

## 日本語Xコミュニティの対策

Xで報告されている実践:

- **70%超えたら即 `/compact`**（自動を待たない派が多数）
- **調査はサブエージェントに全部投げる**（メインの寿命を延ばす）
- **複雑なエラーが出たら「コード修正禁止、状況整理だけ」と指示**（Compaction寸前でのカオスな修正を防ぐ）
- **ミスごとにCLAUDE.mdに1行追加**（compound learning — 次のセッションでも活きる）

特に3番目が面白い。Compactionが近いときにエラーが出ると、Claudeはパニック的に修正を試みて状況が悪化しがち。「直すな、整理しろ」と指示するのが正解。

## PostCompact hookは「まだない」

GitHub Issue #14258でPostCompact hookが要望されている。Compaction直後に自動で特定の指示を再注入する仕組み。

2026年3月時点では**未実装**。Issue #17237でPreCompact/PostCompactの両方が議論中。

これが実装されたら、Compaction後の遵守率低下問題はかなり軽減される。追跡する価値がある。

---

## 自分の対策

自分の環境でやっていること:

- CLAUDE.mdに `Compact Instructions` セクションを配置済み
- `.claude/rules/` で7ファイル分割（CLAUDE.md再読み込みで全ルール生存）
- 調査はサブエージェントに委任
- 30+個のHooksで品質を担保（Compaction後も動作）

Hooksがここでも効く。Hooksはコンテキストに依存しない決定論的な制御だから、Compactionの影響を受けない。「Edit後にPrettier実行」はCompaction後も変わらず動く。

Compaction後に指示を忘れるのはLLMの仕組み上避けられない。でも「忘れても大丈夫な仕組み」は作れる。

## まとめ

| 問題 | 対策 |
|------|------|
| 33Kのバッファが常に予約 | 83.5%トリガーで計算する |
| 83.5%で自動Compaction | 70%で手動 `/compact` |
| Compaction後に遵守率60-70% | Compact Instructions + CLAUDE.mdに永続ルール |
| メインウィンドウの消耗 | サブエージェントに委任 |
| タスク変更時の残骸 | `/clear` |

見えない劣化。スキル予算、Compaction。Claude Codeの問題は「壊れる」ことじゃなく「気づかず劣化する」こと。エラーが出ない分、自分で見に行かないと気づけない。
