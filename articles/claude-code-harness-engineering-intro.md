## 同じAIなのに、順位が28も変わる

Terminal Bench 2.0というベンチマークがある。

Opus 4.6の成績を見て目を疑った。Claude Code内で**33位**。別のエージェントフレームワークに載せ替えると**5位**。同じモデル。同じ重み。変えたのは「ハーネス」だけ。

ハーネスエンジニアリング。この言葉を知ったのは、自分のスキルが80個消えていた事件がきっかけだった。

---

前回の記事で書いた。Claude Codeにスキルを133個作って、80個がモデルから見えなくなっていた話。メタデータの文字数予算を222%オーバーしていた。47個に絞って、全部見えるようになった。

あの整理の途中で「ハーネスエンジニアリング」という概念に出会った。HumanLayerのブログ記事「Skill Issue」。

> 「問題はモデルではなくハーネスにある」

読んで、ちょっと混乱した。スキルの数を増やすほどClaude Codeは賢くなる。そう信じてた。逆だった。

「スキル予算だけじゃなく、ハーネス全体を理解しないとまた同じ轍を踏む」。そう思って、6つのテーマを深掘りした。この記事はその全体地図。

---

## ハーネスって何？

Claude Codeを「AIモデル + 周辺装備」に分けて考える。

- **モデル**: Opus 4.6。これは誰が使っても同じ
- **ハーネス**: CLAUDE.md、スキル、MCP、フック……モデルを「どう使うか」の仕組み全部

33位→5位の差は、ハーネスの差。

レーシングカーに例えるなら、エンジンは同じでシャシーとセッティングが違う。エージェントフレームワークが素のモデルスコアを**10-20ポイント上回る**というデータがある。モデルを1世代アップグレードするより、ハーネスを整えた方が効果が大きいことがある。

---

## 6つの設定レバー

HumanLayerが整理した6つの設定レバー。これが全体地図。

| レバー | 何をするか | コンテキストコスト |
|--------|-----------|------------------|
| **CLAUDE.md** | 常に読み込まれるシステムプロンプト | 常時消費 |
| **MCP Servers** | 外部システムとの接続（GitHub、DB等） | ツール定義で常時消費 |
| **Skills** | ワークフローの手順書 | descriptionだけ常時、本文はオンデマンド |
| **Sub-agents** | メインと別コンテキストのエージェント | 独立（メインを食わない） |
| **Hooks** | ツール実行前後の自動処理 | **ゼロ** |
| **Back-pressure** | テスト・検証による品質担保 | 実行時のみ |

右の列が大事。**コンテキストコスト**。

Claude Codeのコンテキストウィンドウは有限（Opus 4.6で最大1M、以前は200K）。CLAUDE.mdもMCPのツール定義も、存在するだけでトークンを食う。前回のスキル予算問題と同じ構造。「入れれば入れるほど賢くなる」は嘘で、入れすぎると大事な指示を見落とす。

## 各レバーの話

### CLAUDE.md — 全セッションで読まれる指示書

プロジェクトのルートに置く `.md` ファイル。毎回の会話でコンテキストに自動注入される。

**最適な長さ: 50〜150行。**

200行を超えると指示の**30%がtruncation（切り詰め）**される。300行超はむしろ逆効果。Claudeが指示を見落とす頻度が増える。

「全部書きたい」気持ちはわかる。自分もそうだった。答えは `.claude/rules/` への分割。トピック別にファイルを分けると、関連するルールだけが読み込まれる。トークン40%削減、関連性35%向上。

自分のCLAUDE.mdは90行。余裕はある。

### Skills — 呼ばれたときだけ展開される手順書

前回の記事で詳しく書いた。descriptionのメタデータだけが常時コンテキストに入り、SKILL.md本文は呼び出し時にだけ展開される。

Progressive Disclosureという技術を使うと、ある事例では**42,200トークン → 1,900トークン**まで減る。94%削減。CLAUDE.mdを巨大な1ファイルにするのではなく、3層（CLAUDE.md → Skills → References）に分けるだけ。

追加してもベースラインのコストが増えない。O(1)スケーリング。このシリーズの別記事で詳しく書く。

### MCP Servers — 外部システムへの「能力」

MCPとSkillsの違い。ここが一番混乱しやすい。

| 判断基準 | MCP | Skills |
|---------|-----|--------|
| 何か | capability（能力） | orchestration（手順） |
| いつ使う | いつでも・どの文脈でも | 特定ワークフロー時 |
| 例 | GitHubイシュー取得、DB接続 | TDDプロセス、レビューフロー |

判断はシンプル。

1. 「Claudeがいつでも呼べるべき？」→ YES = **MCP**
2. 「繰り返す定型ワークフロー？」→ YES = **Skills**

MCPのツール定義は常にコンテキストを消費する。ただし `defer_loading` で遅延読み込みできる。Skillsにはこのオプションがない（前回のスキル予算問題の根本原因がこれ）。

### Hooks — コンテキストコスト「ゼロ」の最強レバー

6つの中で唯一、コンテキストを**一切消費しない**。

4つのタイプ:

| タイプ | 説明 | 使用率 |
|--------|------|--------|
| command | シェルコマンド実行 | 95% |
| HTTP | Webhookリクエスト | まれ |
| prompt | LLMによるyes/no判定 | 中 |
| agent | ツール付きサブエージェント | まれ |

Exit Codeで挙動が変わるのがミソ:
- **0**: 続行（stdoutが追加コンテキストに）
- **2**: **ブロック**（stderrがフィードバックに）

例えば、ファイル編集後にPrettierを自動実行するHook。1日20回の手動フォーマットコマンドをゼロにできる。

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
      }]
    }]
  }
}
```

自分の環境では30個以上のHookが動いてる。セキュリティチェック、教訓の自動警告、ビルド追跡……全部コンテキストゼロ。正直、一番コスパがいいレバー。

### Sub-agents — コンテキストの分離

メインの会話ウィンドウとは独立したコンテキストでClaudeを動かす。調査やコード検索をサブエージェントに投げると、メインウィンドウが汚れない。

これが重要な理由。Compaction（コンテキスト圧縮）が発動すると、フレームワーク遵守率が**60-70%に落ちる**という報告がある。サブエージェントに重い作業を押し付ければ、メインウィンドウの寿命が延びる。

### Agent Teams — メッシュ型の協調（実験的）

サブエージェントの進化版。通信構造が違う。

- サブエージェント: ハブ＆スポーク。結果をメインに返すだけ
- Agent Teams: **メッシュ**。チームメイト同士が直接通信できる

3チームで3〜4倍のトークンコスト。起動に20-30秒。まだ実験的機能。

適するケース: 独立モジュールの並列開発、仮説の並列テスト。
適さないケース: 同一ファイルの同時編集、密な連携が必要な作業。

## ETH Zurich研究が示す「パラドックス」

ハーネスの話で面白いデータがある。2本の学術論文が真逆の結論を出した。

- **論文A（ETH Zurich）**: AGENTS.mdを入れると成功率**0.5〜2%低下**、コスト**20〜23%増加**
- **論文B**: AGENTS.mdを入れると実行時間**28.64%短縮**、トークン**16.58%削減**

矛盾してるように見えて、実は違う。

AGENTS.mdは**「速くするが、正確にはしない」**。探索範囲を絞って速くなる。でも間違った方向への探索も加速する。

しかもLLMに書かせたAGENTS.mdは効果がマイナス。人間が書いたミニマルなファイルだけが+4%の改善。「AIに設定ファイルを書かせる」のは逆効果という、ちょっと皮肉な結果。

## 自分の環境を振り返ってみた

6つのレバーを知った上で、自分のClaude Code環境をチェックした。

| レバー | 自分の状態 | 判定 |
|--------|-----------|------|
| CLAUDE.md | 90行 | ✅ 最適範囲（50-150行） |
| Skills | 47個 / 予算62% | ✅ 最適化済み |
| MCP | 6台 | ✅ |
| Hooks | 30+個 | ✅ フル活用 |
| Sub-agents | 使用中 | ✅ |
| Agent Teams | 有効化済み | ✅ |
| .claude/rules/ | 7ファイル分割 | ✅ |

全部緑。

……正直、ちょっと拍子抜けした。「何か改善できるはず」と思って研究したのに、すでに整っていた。

でもよく考えると、スキル予算問題をきっかけに整理した一連の作業が、結果的にハーネス全体の最適化になっていたんだと思う。

「たまたま最適化されてた」のと「理由を理解して意図的にやってる」のは全然違う。見えない仕組みを見える化する。それがハーネスエンジニアリング。
